Operational Diagnostics

Troubleshooting

A focused page for recurring system issues and unresolved technical weirdness across OpenClaw, Remote Viewer, Chromebook/Crostini behavior, and related workflow infrastructure. This page now imports the shared Style One package.

Active issue

Cron reliability is still inconsistent

The current understanding is that cron creation and cron delivery/execution are separate questions, and both have shown different failure modes depending on the surface.

  • Telegram-session CLI path has failed with gateway websocket closure.
  • Control/main surface has been able to create jobs more successfully.
  • At least one job appeared to be picked up by the scheduler and then stalled or disappeared in a runningAtMs state without delivering.
Recurring concern

Chromebook / Crostini startup behavior

There is still an unresolved concern around intermittent Chromebook/Crostini terminal startup delay and overall environment readiness after reboot.

  • The working theory has leaned more toward Crostini/container startup state than a proven permanent OpenClaw failure.
  • Connection stability across workdays and travel has often been good once running.
  • The real fragility may be less about network and more about device/container active-use state.
Recent bug lane

Fullscreen / input quirks on Chromebook

The now-removed shared floating navigation exposed a narrower interaction bug where touchscreen behavior and mouse/trackpad behavior did not match cleanly on Chromebook.

  • Touch input was able to trigger fullscreen more reliably than mouse/trackpad click.
  • Mouse/trackpad clicks were being interfered with by the drag logic in the floating control.
  • This reinforced a useful lesson: shared UI abstractions can hide input-specific bugs.
Newly noticed

Memory search path seems degraded

The raw memory files are readable, but the formal memory_search retrieval layer returned weak or empty results even for queries that should plausibly have matched known content.

  • Direct reading of MEMORY.md and memory/*.md still works.
  • The search layer previously reported provider: none and mode: fts-only.
  • That suggests discovery/indexing is weaker than raw access, not that memory itself is missing.
Known tool-handoff issue

Browser relay connected, but browser control may not reach this session

The browser relay/extension can appear successfully installed and connected on Christopher’s side while this active chat session still fails to receive usable browser-control capability.

  • The likely failure point is tool handoff into the active session, not necessarily the relay connection itself.
  • This means “relay connected” should not be treated as proof that browser control is actually available here.
  • The issue seems to live in the gap between browser-side connection and session-side tool exposure.
Why this page exists: to make active technical problems easier to revisit without relying on scattered chat memory. If this lane keeps proving useful, it can later grow into a fuller troubleshooting hub or connect more explicitly to the incident-reports archive.
