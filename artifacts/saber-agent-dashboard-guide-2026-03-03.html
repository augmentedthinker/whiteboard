⚔️ Saber Agent Dashboard Guide (Expanded)

A practical explanation of what each section means and what it actually allows Saber to do.

What you’re looking at

The Agents page is the control room for one agent identity. It determines:

  • which instructions shape behavior (Files),
  • what actions are allowed (Tools),
  • which specialized instruction packs are available (Skills),
  • how messaging/scheduling context applies (Channels/Cron).

Important: main is the internal ID key. Saber is the human-facing identity name.

1) Overview tab

  • Shows workspace path, identity metadata, default status, and model stack.
  • Primary model and fallback model(s) are configured here.
  • This tab is your high-level runtime posture: who the agent is + which brain it uses.

2) Files tab

  • Shows core behavioral files from the workspace (AGENTS/SOUL/TOOLS/IDENTITY/USER/HEARTBEAT/MEMORY).
  • These files are your “operating constitution.”
  • Changing these changes agent behavior more deeply than changing model alone.

3) Tools tab (what each permission actually means)

Live status: 25/25 enabled, profile full, source default.

Files permissions

  • read: can open files and inspect content.
  • write: can create/overwrite files.
  • edit: can make exact-string surgical edits.
  • apply_patch: can apply structured diff-style patches.

Effect: lets Saber document, refactor, and ship artifacts in your workspace.

Runtime permissions

  • exec: run shell commands (git, openclaw CLI, scripts).
  • process: monitor long-running exec sessions.

Effect: real system operations, diagnostics, and automation execution.

Web permissions

  • web_search: external search provider queries (when configured).
  • web_fetch: fetch readable content from URLs.

Effect: current-event research and source retrieval.

Memory permissions

  • memory_search: semantic lookup across MEMORY lanes.
  • memory_get: read selected lines from memory files.

Effect: continuity, decisions recall, and anti-repetition context.

Session/agent orchestration permissions

  • sessions_list/history/send: inspect and message other sessions.
  • sessions_spawn, subagents: start/manage isolated runs.
  • session_status, agents_list: operational visibility.

Effect: multi-session control and delegation when needed.

UI + Messaging + Automation + Media

  • browser/canvas: automate and inspect web UI surfaces.
  • message/tts/image: outbound communication and media handling.
  • cron/gateway/nodes: scheduling, gateway controls, and paired device operations.

Quick Presets

  • Minimal: narrow safe surface.
  • Coding: file/runtime heavy, less messaging emphasis.
  • Messaging: communication-focused lane.
  • Full: broad capability (current mode).
  • Inherit: follow upstream defaults.

4) Skills tab (what “51/51 enabled” means)

  • Skills are packaged instruction modules (task-specific playbooks).
  • 51/51 enabled means Saber currently inherits all available built-in skills from your installation.
  • If you disable some, this tab becomes a strict allowlist (only checked skills are usable).

In your active setup, relevant examples include skills like GitHub ops, weather, healthcheck, session-logs, and custom skill tooling.

Practical rule: keep all enabled while exploring; narrow allowlist when you want a strict production behavior profile.

5) Channels tab

  • Shows gateway channel status in this agent context (e.g., Telegram connected state).
  • Exposes policy flags (DM/group policy) that explain routing behavior.
  • Use this first when a message “should have sent but didn’t.”

6) Cron Jobs tab

  • Shows scheduler health and jobs assigned to the selected agent.
  • Current state in your setup: No Saber-assigned jobs.
  • You may still see gateway-level scheduler stats unrelated to Saber jobs.

How to program Saber intentionally (recommended pattern)

  1. Write operating rules in AGENTS.md and persona/tone in SOUL.md.
  2. Keep tools at Full during build phases; tighten presets before high-risk automation.
  3. Use Skills tab to constrain behavior only when you need reproducible, narrow task lanes.
  4. Keep one active agent and minimal cron by default (your current posture).
  5. Capture experiments in Ideas/OpenClaw Hub before turning them into scheduled systems.

Bottom line

You now have a clean single-agent architecture: Saber is the operator, tools are fully enabled, skills are fully available, and scheduling is intentionally sparse. This is the right base for controlled capability growth without automation drift.
