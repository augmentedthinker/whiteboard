What you’re looking at

The Agents page is where you configure how Saber behaves: identity, model, core files, tools, skill allowlist, channel context, and cron relationship.

Important: main is the internal technical ID. Saber is the display identity name.

1) Overview tab

  • Shows workspace path, identity metadata, default status, and active model stack.
  • Primary model + fallback model can be edited here.
  • Use this tab when you want to change behavior globally without touching tool toggles.

2) Files tab

  • Shows the core control files Saber reads from your workspace.
  • Current key files visible: AGENTS.md, SOUL.md, TOOLS.md, IDENTITY.md, USER.md, HEARTBEAT.md, MEMORY.md.
  • This is your “program the operator” panel — edit these and you change how Saber thinks/acts.

3) Tools tab (most important)

Live status observed: 25/25 enabled, profile full, source default.

  • Files: read, write, edit, apply_patch
  • Runtime: exec, process
  • Web: web_search, web_fetch
  • Memory: memory_search, memory_get
  • Sessions/agents: sessions_list/history/send/spawn, subagents, session_status, agents_list
  • UI: browser, canvas
  • Messaging/media: message, tts, image
  • Automation/system: cron, gateway, nodes

Quick Presets are your fast safety modes: Minimal / Coding / Messaging / Full / Inherit.

4) Skills tab

  • Per-agent skill allowlist.
  • Live snapshot showed 51/51 enabled.
  • If you disable some skills, this tab becomes your precise specialization filter.

5) Channels tab

  • Channel connectivity/status snapshot for this agent context.
  • You can verify Telegram health + policy flags (DM/group policy) here.
  • Use this when message delivery feels inconsistent.

6) Cron Jobs tab

  • Shows scheduler health and jobs assigned specifically to Saber.
  • Current state: No jobs assigned to Saber.
  • You may still see global scheduler job counts from gateway-level context.

How to “program” Saber intentionally

  1. Define operating behavior in AGENTS.md + SOUL.md.
  2. Constrain or expand action surface in Tools tab presets.
  3. Use Skills as a capability allowlist, not a dumping ground.
  4. Keep defaults simple: one primary model, one fallback, one active agent.
  5. Use Ideas/Lose Ends lane for future systems before turning them into cron.

Bottom line

You now have a clean single-agent architecture: Saber is the operator, tools are fully enabled, no per-agent cron jobs, and behavior is driven by your workspace control files. This is exactly the right base for building agents intentionally instead of letting automation sprawl.
