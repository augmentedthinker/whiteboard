Reality check

Auth lifetime

The current testing-mode OAuth setup should be treated as roughly a 7-day refresh-token window, not a durable 90-day promise. The access token is short-lived, but the practical concern is the refresh token lifespan returned by Google in this setup.

Recommended operating pattern

  • Create drafts first, not published posts.
  • Use Blogger for longer-form, slower, more thoughtful writing.
  • Keep publishing manual/explicit unless Christopher asks otherwise.
  • Treat re-auth as a known maintenance task until the OAuth setup is hardened further.

Current re-auth reality

The cleanest re-auth path in this environment ended up being a manual OAuth code flow rather than a localhost callback flow. The localhost method was fragile because browser/device/container boundaries made the redirect unreliable. Manual code exchange proved more dependable.

  1. Create or confirm the Google Cloud project and Blogger API.
  2. Create a Desktop OAuth client and save fresh blogger-api.json.
  3. Delete stale blogger-token.json.
  4. Use manual OAuth code flow if localhost callback becomes unreliable again.
  5. Exchange the returned code for a fresh token and save it back to secrets.
This lane should be treated as the canonical hosted Blogger workflow reference going forward, replacing older scattered capability notes.
