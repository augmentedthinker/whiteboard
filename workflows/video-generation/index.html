Veo Video Generation

This page is the browser-readable backup playbook for creating AI-generated videos from the main session using Gemini / Veo. It is designed to be complete enough that a future session could reconstruct the workflow from this page alone, even if the local markdown file were unavailable.

Current status Requirements Core route Step-by-step Shell template Failure modes Verification

Current status

Status: proven working from the main session as of 2026-03-30.

  • Video generation works through Gemini’s long-running Veo endpoint.
  • The reliable model currently documented here is veo-3.0-fast-generate-001.
  • The successful path is not the browser checker and not a simple one-shot generateContent call.
  • The workflow is: start a long-running prediction, poll the operation until it is done, extract the returned file URL, and then download the MP4 with the API key header.

Requirements

  • Shell access from a session that can run curl and python3.
  • A readable Gemini API key. Current preferred key path in the local system is:
    /mnt/chromeos/MyFiles/Downloads/shared_workspace/secrets/GEMINI_API_KEY
  • An output folder in shared workspace, typically:
    /mnt/chromeos/MyFiles/Downloads/shared_workspace/generated-videos/YYYY-MM-DD/
  • Enough patience to poll the operation — the first response only gives an operation name, not the final video.

Important distinction

The browser-side Veo checker is still useful for diagnostics, but the main-session CLI long-running route is the real proven path.

Core route

Model

veo-3.0-fast-generate-001

Start endpoint

POST https://generativelanguage.googleapis.com/v1beta/models/veo-3.0-fast-generate-001:predictLongRunning?key=GEMINI_API_KEY

Poll endpoint pattern

GET https://generativelanguage.googleapis.com/v1beta/<operation-name>?key=GEMINI_API_KEY

Result extraction path

response.generateVideoResponse.generatedSamples[0].video.uri

Download rule

Use the returned URI with the header:
x-goog-api-key: GEMINI_API_KEY

Step-by-step procedure

  1. Read the API key from the secrets location and strip trailing newlines.
  2. Create a dated output folder in shared workspace.
  3. Write a request JSON using the shape {"instances":[{"prompt":"..."}],"parameters":{"aspectRatio":"16:9"}}.
  4. POST to predictLongRunning and save the response JSON.
  5. Extract the operation name from the start response.
  6. Poll the operation every ~10 seconds until done: true.
  7. Extract the nested video URI from the completed response.
  8. Download the MP4 with the x-goog-api-key header.
  9. Write sibling metadata beside the video so future sessions know what was generated and how.

Proven request shape

{ "instances": [ { "prompt": "A cinematic 6-second shot of a lone traveler standing on a rooftop in Washington, D.C. at blue hour, soft wind moving their coat, city lights glowing below, subtle camera dolly forward, tasteful sci-fi atmosphere, realistic, elegant, no text, no watermark." } ], "parameters": { "aspectRatio": "16:9" } }

Reusable shell template

This is the compact template version of the workflow. It is not the only valid implementation, but it captures the proven shape clearly enough to recreate the route.

set -euo pipefail KEY_FILE="/mnt/chromeos/MyFiles/Downloads/shared_workspace/secrets/GEMINI_API_KEY" API_KEY="$(tr -d '\r\n' < "$KEY_FILE")" OUT_DIR="/mnt/chromeos/MyFiles/Downloads/shared_workspace/generated-videos/$(date +%F)" mkdir -p "$OUT_DIR" PROMPT='REPLACE WITH VIDEO PROMPT' BASENAME='replace-with-output-name' cat > "$OUT_DIR/request.json" <<'JSON' { "instances": [ { "prompt": "REPLACE WITH VIDEO PROMPT" } ], "parameters": { "aspectRatio": "16:9" } } JSON START_URL="https://generativelanguage.googleapis.com/v1beta/models/veo-3.0-fast-generate-001:predictLongRunning?key=${API_KEY}" curl -sS -X POST "$START_URL" \ -H 'Content-Type: application/json' \ --data @"$OUT_DIR/request.json" \ -o "$OUT_DIR/start-response.json" OP_NAME=$(python3 - <<PY import json with open("$OUT_DIR/start-response.json") as f: print(json.load(f)["name"]) PY ) STATUS_URL="https://generativelanguage.googleapis.com/v1beta/${OP_NAME}?key=${API_KEY}" for i in $(seq 1 40); do curl -sS "$STATUS_URL" -o "$OUT_DIR/op-status.json" DONE=$(python3 - <<PY import json with open("$OUT_DIR/op-status.json") as f: data=json.load(f) print('true' if data.get('done') else 'false') PY ) if [ "$DONE" = "true" ]; then break fi sleep 10 done VIDEO_URL=$(python3 - <<PY import json with open("$OUT_DIR/op-status.json") as f: data=json.load(f) print(data["response"]["generateVideoResponse"]["generatedSamples"][0]["video"]["uri"]) PY ) VIDEO_PATH="$OUT_DIR/${BASENAME}.mp4" curl -L -sS -H "x-goog-api-key: ${API_KEY}" "$VIDEO_URL" -o "$VIDEO_PATH" echo "Saved to: $VIDEO_PATH"

Practical note

In actual sessions, it can be safer to write the final prompt literal directly into the request JSON rather than over-generalizing the shell template too early.

Common failure modes

  • Missing API key: the key file is absent, unreadable, or malformed.
  • Wrong endpoint: trying generateContent instead of predictLongRunning.
  • Wrong model: swapping away from the currently proven veo-3.0-fast-generate-001 without testing.
  • Stopping too early: forgetting that the first response is only an operation handle.
  • Bad extraction: failing to pull generatedSamples[0].video.uri from the completed response.
  • Download auth mistake: not sending the x-goog-api-key header on the final file download.
  • Surface mismatch: a remote session may need delegation if it lacks shell access or workspace access.

Verification standard

Treat the workflow as successful only if all of these are true:

  1. the start request returns an operation name
  2. polling eventually returns done: true
  3. a video.uri appears in the completed response
  4. the MP4 downloads successfully
  5. the file exists in shared workspace
  6. the output path is clearly reported
  7. ideally, a metadata markdown file is written beside it

Current truth

The point now is no longer proving that video generation is possible. The point is preserving the route clearly enough that future sessions can do it again without guesswork.
